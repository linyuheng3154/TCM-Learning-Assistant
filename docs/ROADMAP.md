# TCM-Learning-Assistant 项目迭代需求规划

---

## 一、数据层需求

### D1. 方剂数据库扩充
**目标**: 从10首扩展至100首经典方剂

**数据来源优先级**:
1. 中国药典(2020版)收录方剂
2. 《伤寒论》《金匮要略》经方
3. 《太平惠民和剂局方》名方
4. 现代临床常用方剂

**验收标准**:
- 至少100首方剂完整数据
- 每首方剂包含完整的组成、功效、主治、出处
- 数据通过JSON Schema校验

---

### D2. 药材-方剂双向索引
**目标**: 实现药材与方剂的双向查询

**API接口设计**:
```
GET /herbs/{herb_name}/formulas     # 获取含该药材的方剂
GET /herbs/{herb_name}/stats        # 药材使用统计
GET /formulas/{id}/herbs            # 获取方剂药材详情
```

**验收标准**:
- 双向查询响应时间 < 100ms
- 药材数据覆盖方剂中出现的所有药材

---

### D3. 现代研究数据集成
**目标**: 为传统方剂补充现代医学研究数据

**数据类型**: 药理研究、临床试验、现代应用

**验收标准**:
- TOP20高频方剂补充现代研究数据
- 数据来源可追溯

---

## 二、技术层需求

### T1. 数据存储迁移 (JSON → SQLite)
**目标**: 提升查询性能和数据管理能力

**迁移策略**:
1. 创建SQLite数据库和表结构
2. 编写JSON→SQLite迁移脚本
3. 更新Service层使用SQLite
4. 保留JSON作为备份/导入格式

**验收标准**:
- 所有JSON数据完整迁移
- 查询性能提升 > 50%

---

### T2. 智能搜索增强
**目标**: 提升搜索准确性和用户体验

**功能清单**: 拼音搜索、模糊匹配、症状映射、权重排序

**验收标准**:
- 拼音首字母搜索准确率 > 90%
- 症状搜索返回相关方剂
- 搜索响应时间 < 200ms

---

### T3. 缓存层实现
**目标**: 优化高频查询性能

**验收标准**:
- 热点数据缓存命中率 > 80%
- 缓存失效后数据一致性保证

---

## 三、功能层需求

### F1. AI辨证推荐系统
**目标**: 根据用户症状智能推荐方剂

**系统流程**: 用户输入症状 → 症状解析 → 证型判断 → 方剂推荐 → 结果解释

**验收标准**:
- 常见证型识别准确率 > 85%
- 推荐结果包含合理性解释
- 明确标注AI推荐免责声明

---

### F2. 方剂对比功能
**目标**: 支持两首或多首方剂的对比分析

**对比维度**: 组成对比、功效对比、适应症对比、禁忌对比

**验收标准**:
- 支持最多5首方剂同时对比
- 自动生成对比摘要

---

### F3. 方剂加减变化
**目标**: 展示方剂的加减变化规律

**验收标准**:
- 收录经典加减变化
- 支持变化关系可视化

---

## 四、工程层需求

### E1. 单元测试完善
**目标**: 测试覆盖率达到80%以上

### E2. Docker容器化
**目标**: 简化部署流程

### E3. API限流保护
**目标**: 防止API滥用

---

## 优先级排序

| 优先级 | 需求 | 理由 |
|--------|------|------|
| P0 | D1 数据扩充 | 基础数据量不足 |
| P0 | T1 SQLite迁移 | 性能瓶颈 |
| P1 | F1 AI辨证推荐 | 核心价值功能 |
| P1 | T2 搜索增强 | 用户体验 |
| P2 | D2 双向索引 | 功能完善 |
| P2 | E1 测试完善 | 工程质量 |
| P3 | 其他 | 迭代优化 |
